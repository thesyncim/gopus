# gopus v0.1.0

Draft status: this is not a published release until the `v0.1.0` tag and the
matching GitHub Release both exist.

## Highlights

- Pure Go Opus encode/decode with no `cgo` or external C toolchain
- caller-owned zero-allocation `Encode` / `Decode` hot path
- Ogg Opus container support via `container/ogg`
- multistream encode/decode support
- libopus-relative parity, provenance, and release-evidence workflow

## Stable Core

The intended stable core for `v0.1.0` is:
- `Encoder`
- `Decoder`
- `MultistreamEncoder`
- `MultistreamDecoder`
- `container/ogg`
- caller-owned `Encode` / `Decode` hot-path APIs

## Supported Default-Build Surface

The default build is the supported public contract for `v0.1.0`.

## Optional Extension Contract

Supported optional controls in the default build:
- `SetDNNBlob(...)` on `Encoder`, `Decoder`, `MultistreamEncoder`, and `MultistreamDecoder`

`make test-dnn-blob-parity` validates the default `SetDNNBlob(...)` control surface against pinned libopus USE_WEIGHTS_FILE model blobs and fails on skipped helper coverage. Decoder-side `SetDNNBlob(...)` currently covers loader-derived validation and retained control state. In DRED-tagged or quarantine builds, `SetDNNBlob(...)` may also bind DRED-capable model families on the control path, but normal encode/decode runtime work remains dormant until a DRED duration, payload, or recovery path is explicitly armed. Model-only public caller-buffer encode/decode paths stay zero-allocation and skip unarmed DRED helper work. Full model-backed PLC/OSCE runtime behavior remains incomplete in `v0.1.0`.

Unsupported or tag-gated in the default build:
- QEXT controls are supported only when built with `-tags gopus_qext`, which exposes `SetQEXT(...)` / `QEXT()` on `Encoder` and `MultistreamEncoder`; `make test-qext-parity` enforces the supported QEXT packet-extension seams against a separate pinned libopus build configured with `--enable-qext` and fails on skipped libopus-helper coverage, while default builds keep QEXT controls absent and do not scan, arm, or encode QEXT packet extensions in public hot paths
- DRED control and standalone surfaces are supported only when built with `-tags gopus_dred`, which exposes `SetDREDDuration(...)` / `DREDDuration()` and standalone `DREDDecoder` / `DRED` helpers; the internal encoder DRED runtime, top-level decoder DRED internals, and multistream decoder DRED cache/runtime helpers are build-tag split out of default `.`, `./encoder`, and `./multistream` builds; standalone DRED wrapper lifecycle/no-allocation, libopus parse/decode/process metadata checks, real-packet standalone process state/feature parity, standalone recovery scheduling parity, decoder cached recovery bookkeeping parity, SILK wideband 20/40/60 ms mono and 20 ms stereo carried-payload/primary-frame seams, the Hybrid fullband 20 ms payload-only seam, and the SILK 20 ms primary-budget seam are enforced under `make test-dred-tag`; quarantine API exposure, standalone/control smoke, cached bookkeeping, and dormant-runtime checks are pinned by `make test-unsupported-controls-tag`; those encoder seams plus parser availability, internal converter/payload/basic-analysis coverage, real-model PitchDNN and RDOVAE encoder oracles, the conceal-analysis oracle, 48 kHz bootstrap coverage, and the current mono decoder explicit/live numerical matrix are mirrored by `make test-unsupported-controls-parity`; required DRED parity gates fail on skipped libopus-helper tests, while Hybrid packet-shape smoke remains outside supported-tag claims and Hybrid primary-frame byte exactness plus broader stereo/multistream decoder support remain unsupported/open
- Supported tags may be combined: `-tags "gopus_dred gopus_qext"` reports DRED and QEXT support, while `-tags "gopus_unsupported_controls gopus_qext"` reports QEXT support and leaves DRED/OSCE BWE unsupported despite exposing quarantine controls
- `SetOSCEBWE(...)` / `OSCEBWE()` are absent unless built with `-tags gopus_unsupported_controls`
- The `gopus_unsupported_controls` build remains a parity/quarantine umbrella and does not itself make `SupportsOptionalExtension(...)` report feature support

The `gopus_unsupported_controls` build may expose DRED controls and standalone
helpers for quarantine parity, but release support still comes only from
`gopus_dred`.

Low-level implementation packages remain pre-v1 and may change more freely than the top-level `gopus` facade plus `container/ogg`.

## Support Matrix

- Go: `1.25+`
- Tested in CI: Linux, macOS, Windows
- Optimized kernels: `amd64` and `arm64`
- Pure-Go fallbacks: available on other architectures

## Known Limits

- This is the first tagged public release and remains pre-v1.
- Some optional libopus build-time extensions remain intentionally unsupported in the default build.
- Low-level/internal-facing packages may continue to change faster than the stable core.
