# Supply Chain Controls

This document records controls that users can verify from the repository and
release artifacts.

## Dependency Updates

- Dependabot is enabled in `.github/dependabot.yml` for GitHub Actions, the root
  Go module, `examples/webrtc-control`, and the external consumer smoke module.
- Go module changes are reviewed through normal PR checks.
- Workflow and release-critical file changes are covered by `.github/CODEOWNERS`.

## GitHub Actions Permissions

Workflows use explicit permissions:

| Workflow | Permission shape |
| --- | --- |
| `CI` | `contents: read` |
| `Verify Safety` | `contents: read` |
| `Verify Production Exhaustive` | `contents: read` |
| `OpenSSF Scorecard` | `contents: read`, `security-events: write`, `id-token: write` |
| `Release` | `contents: write` only so it can create/update GitHub Releases and upload release evidence |

Any workflow that needs broader permissions must document why in the workflow and
pass CODEOWNERS review.

## Action Version Policy

Existing workflows use version tags such as `actions/checkout@v5` instead of
commit SHAs for now. The tradeoff is intentional for the first release: tagged
updates let Dependabot surface upstream security and compatibility fixes without
manual SHA refresh work.

Review expectations for every action update:

- Verify the action repository and release notes.
- Check whether requested permissions changed.
- Confirm the workflow still uses the least privilege needed for its job.
- For new third-party actions in release-critical workflows, decide during review whether the action should be pinned by full commit SHA instead of a version tag.

## Scorecard

`.github/workflows/scorecard.yml` runs OpenSSF Scorecard on `master`, on a weekly
schedule, and on manual dispatch. Results are uploaded as SARIF so security
signals are visible in GitHub code scanning.

## Release Provenance And SBOM Plan

For `v0.1.0`, the release evidence bundle generated by `make release-evidence`
is the required provenance artifact. It records:

- Release tag and commit SHA.
- Go version and platform.
- `go.mod`, `go.sum`, `Makefile`, `tools/ensure_libopus.sh`, and libopus tarball
  SHA256 checksums.
- Pinned libopus version and expected tarball SHA256.
- Command manifest, command logs, and pass/fail summaries.
- `go list -m -json all` module inventory.

The first source-only release does not ship built binaries. If future releases
publish binaries, add signed checksums and SLSA provenance for those binaries.
Before adding runtime dependencies, pin a formal SPDX or CycloneDX generator and
attach its SBOM output alongside the release evidence archive.
